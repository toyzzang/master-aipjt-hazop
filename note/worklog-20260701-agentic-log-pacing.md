# 작업 설계: Agentic 실시간 로그 개선

## 배경

현재 실시간 로그는 초안생성 버튼을 누르면 너무 빠르게 출력된다.
사용자 입장에서는 AI Agent가 실제로 자료를 읽고 판단하는 느낌보다, 서버가 정해진 상태 메시지를 한 번에 찍는 느낌이 강하다.

또한 MSDS 조회 내부 흐름인 `KOSHA 조회 -> Bing fallback -> 내장 요약 fallback`이 화면 로그에 충분히 드러나지 않는다.

## 설계 판단

LLM에게 모든 로그 문구를 매번 생성시키는 방식은 가능하지만 1차 PoC에서는 비효율적이다.

이유:

- 로그 한 줄마다 LLM 호출을 하면 속도와 비용이 커진다.
- 로그 생성 실패가 본 작업 실패로 이어질 수 있다.
- 로그는 사실성과 재현성이 중요하므로, 실제 수행한 Tool 단계에 맞춰 시스템이 근거 있는 문구를 만드는 편이 안전하다.

따라서 이번 PoC에서는 다음 방식을 선택한다.

- 실제 Tool 수행 결과를 기반으로 Agent 스타일의 로그 문구를 생성한다.
- 로그 문구는 고정 상태값이 아니라, 입력 물질/조회 결과/fallback 사유를 포함한다.
- 로그 출력 사이에 짧은 지연을 두어 사용자가 진행 상황을 따라갈 수 있게 한다.
- LLM 본 생성은 `#3`, `#4` 구조화 초안 생성에 집중한다.

## 수정 범위

1. MSDS 조회 함수가 최종 요약뿐 아니라 조회 trace를 반환하도록 추가한다.
2. Agent 실행 로그에 아래 내용을 표시한다.
   - KOSHA 검색 시작
   - KOSHA 검색 성공/실패
   - KOSHA 결과 부정확 판단
   - Bing fallback 시도/생략
   - 내장 MSDS 요약 사용
3. Agent 로그 출력 사이에 configurable delay를 추가한다.
4. Docker/환경 변수에 로그 지연 설정값을 추가한다.

## 비수정 범위

- 위험도 계산 기준은 변경하지 않는다.
- Excel 구조는 변경하지 않는다.
- LLM Provider는 변경하지 않는다.
- DB 스키마는 변경하지 않는다.

## 성공 기준

- 버튼 클릭 후 로그가 한 번에 몰려 나오지 않는다.
- MSDS 조회 경로가 화면 로그에 단계별로 나온다.
- KOSHA 성공 시 출처와 chem_id가 표시된다.
- KOSHA 실패 후 내장 요약 사용 시 fallback 사유가 표시된다.
- 샘플 3개 demo 흐름이 계속 통과한다.

## 구현 결과

- `fetch_msds_summary_with_trace()`를 추가해 MSDS 최종 요약과 조회 trace를 함께 반환한다.
- Agent는 KOSHA 조회, Bing fallback, 내장 요약 fallback을 각각 별도 로그로 표시한다.
- `AGENT_LOG_DELAY_SECONDS` 환경 변수로 로그 pacing을 조정한다.
