# HAZOP AI Agent 최종 시연영상 촬영 가이드

> 상태: 9차 피드백 반영 v10 최종 촬영·편집 기준

## 최종 우선 적용 규칙

- Excel의 #1·#2 내용을 별도 캡처 화면으로 보여주지 않는다. 실제 브라우저에서 파일을 업로드하는 동작만 보여준다.
- 영상의 0초 첫 프레임부터 완성된 시작 표지가 보여야 한다. 녹화 준비 화면이나 브라우저 화면이 먼저 보이는 프레임은 모두 잘라낸다.
- 모든 자막은 하단 전용 여백의 폭 1770px 고정 바에 표시한다. 같은 번호의 두 문장은 한 자막에서 두 줄로 동시에 보여준다.
- 자막은 1920×1080 기준 38px 흰색 굵은 글씨이며, 반투명 진회색 바의 정중앙에 배치한다.
- 스피너 회전, 로그 이동, 실시간 갱신처럼 화면에서 바로 보이는 동작은 자막으로 설명하지 않는다.
- AI 초안생성 직후 전체 흐름을 `위험성평가 초안 작성 Agent → 검토 Agent → 조치계획 Agent` 순서로 한 번 설명한다.
- 확대 로그는 `공통 MSDS 근거 조회 → 위험성평가 초안 작성 Agent 시작 → Planning → Skill → Tool → 작성 → 검토 Agent 시작 → Self-Correction → 위험도 선별 → 조치계획 Agent 시작` 순서로만 보여준다.
- 설명을 위해 이미 지나간 상단 로그로 되돌아가지 않는다. 강조할 로그는 생성 순서대로 한 방향으로 이동한다.
- 중요 로그는 5px 진한 청록색 둥근 테두리만 사용해 강조하고 배경색과 그림자는 넣지 않는다.
- 클릭 표시는 `#006b4f` 진한 초록색 7px 원형 테두리로 표시한다.
- 7차 최종본에서는 강조 박스의 배경색을 제거하고 5px 청록색 둥근 테두리만 사용한다.
- Planning 설명은 상단 고정 헤더가 아니라 실제 계획 항목이 적힌 `Planning 실행 현황` 로그 박스 전체만 강조한다.
- 작성 Agent 설명은 `Tool 결과를 취합해 Agent 결과를 생성 중입니다.` 로그 박스를 중앙에 배치하고 그 박스만 강조한다. `Context Prompt를 구성했습니다.` 로그는 강조하지 않는다.
- 검토 Agent와 Self-Correction 설명도 Agent 전체 카드가 아니라 자막이 설명하는 실제 실행 로그 박스 하나만 강조하며, 해당 박스를 가능한 한 화면 중앙에 배치한다.
- 로그 영역의 높이, padding, margin은 애플리케이션 원본 값을 그대로 사용한다. 로그를 중앙에 보이게 하려고 인위적인 아래 여백을 만들지 않고 원본 로그 영역 안의 스크롤 위치만 조정한다.
- LLM 응답 대기는 배속하지 않는다. 실행 모습을 잠시 보여준 뒤 대기 중간을 편집하고 다음 Step부터 1.0배속으로 잇는다.
- 결과 자막은 #3 표, 근거 열, #4 표, 검토표가 각각 화면에 보이는 동안에만 해당 내용을 설명한다.
- 자막 글씨에는 외곽선과 그림자를 사용하지 않고 순수 흰색으로 표시하며, 하단 고정 바의 수직·수평 정중앙에 배치한다.
- 자막은 대응 화면이 먼저 보인 뒤 약 0.12초 후 시작한다. 화면보다 자막이 먼저 나오는 편집을 허용하지 않는다.
- 위험도 9 이상 고위험 결과는 실제 실행값 2~4건이면 정상으로 인정한다. 3건으로 고정하거나 영상에서 숫자를 조작하지 않는다.
- 아래의 과거 타임라인이나 문구와 충돌할 경우 이 최종 우선 적용 규칙을 따른다.
>
> 목표 길이: 3분 50초~4분 05초, 권장 완성본 4분 03초
>
> 시연 샘플: `HAZOP_ASM_Epsilon3200_10Guidewords.xlsx`

## 1. 절대 변경하지 않는 제작 원칙

- 모든 자막은 하단 중앙의 자막 전용 여백에 배치한다. 영상 내용 위로 자막을 옮기지 않는다.
- 모든 자막은 동일한 글꼴, 크기, 흰색 글자, 반투명 진회색 배경을 사용한다. 장면에 따라 크기나 디자인을 바꾸지 않는다.
- 시작 표지에는 `실제 브라우저`, `실제 엑셀`, `실제 에이전트 로그 시연` 문구를 절대 넣지 않는다.
- 시작 표지 다음에 흰 화면을 끼우지 않고 `1. 데이터 입력` 표지로 바로 전환한다.
- 시작·목차·종료 표지는 v3 영상과 같은 짙은 남색 고정 배경, 흰색 제목, 청록색 포인트로 통일한다.
- 표지에는 번쩍임, 입자, 글로우, 줌, 필름 노이즈, 움직이는 배경을 사용하지 않는다.
- 웹 화면은 아래쪽에 자막 전용 여백을 남겨 버튼, 로그, 결과 그리드가 자막에 가려지지 않게 한다.
- 마우스 포인터를 항상 보이고 클릭 순간에는 포인터 주변에 짧은 원형 표시를 넣는다.
- Agent 로그는 탭별로 필터링하지 않고 `전체` 상태로 본다.
- 로그 설명을 위해 영상을 정지하거나 0.5초씩 끊어 이동하지 않는다. 생성된 로그를 순서대로 자연스럽게 따라간다.
- 긴 LLM 대기 구간은 시작 부분을 잠시 보여준 뒤 대기 중간을 잘라내고 다음 로그가 나타나는 정상 속도 장면으로 잇는다.
- Planning, Tool, Context, Self-Correction 로그는 청록색 또는 초록색 테두리만 사용해 강조한다.
- 빨간 테두리는 사용하지 않는다. `#3 위험성평가`, `#4 조치계획서`, 근거 열도 초록색 테두리로 강조한다.
- 영상 끝에는 이름과 사번 없이 `시연을 마치겠습니다`와 `감사합니다.`를 표시한다.
- 같은 편집본에서 자막 레이어만 켜고 끄는 방식으로 자막 포함본과 무자막본을 각각 출력한다.

## 2. 표지 디자인

### 공통 디자인 · v3 스타일 고정

- 배경: 짙은 남색 또는 차콜 `#1E232D` 계열 단색.
- 포인트: 청록색 `#25D6C8` 계열.
- 제목: 흰색 굵은 글씨, 화면 중앙 정렬.
- 보조문구: 연한 회색, 제목 아래 중앙 정렬.
- 하단 장식: 짧은 청록색 수평선.
- 모든 요소는 고정한다. 전환은 짧은 디졸브만 사용한다.

### 시작 표지 · 5초

```text
AI MASTER PROJECT 6기
AI 기반 공정위험성평가 HAZOP 초안 자동 생성
김창은 · 10346
```

표지에 넣지 않을 문구:

- 실제 브라우저
- 실제 엑셀
- 실제 에이전트 로그 시연
- PPT 기반 시연
- FAST FORWARD

### 목차 표지 · 각 3초

```text
SECTION 01
1. 데이터 입력
설비 정보와 평가 대상 등록
```

```text
SECTION 02
2. Agent 실행
Planning부터 Self-Correction까지
```

```text
SECTION 03
3. 결과 확인
판단 근거 · 조치계획 · 초안 검토 및 보완 내역
```

### 종료 표지 · 5초

```text
DEMO COMPLETE
시연을 마치겠습니다
감사합니다.
```

- 종료 표지에는 이름과 사번을 넣지 않는다.
- 문구는 `시연을 마치겠습니다`로 고정한다.

## 3. 자막 공통 규칙

- 위치: 화면 하단 중앙, 하단에서 약 70px 위.
- 크기: 1920×1080 기준 38px 고정.
- 글꼴: Pretendard 또는 Noto Sans KR SemiBold 고정.
- 글자색: 흰색.
- 배경: 진회색 `#20242A`, 불투명도 78%, 모서리 12px.
- 자막 영역: 브라우저 아래쪽에 별도 여백을 만들고, 영상 내용과 겹치지 않게 한다.
- 자막 바 폭: 1770px. 같은 번호에 `<br>`로 묶인 두 문장은 하나의 자막에서 두 줄로 동시에 표시한다.
- 자막이 길어도 `작성`과 `합니다`처럼 서술어를 억지로 줄바꿈하지 않는다.
- 한 자막에는 한 가지 핵심만 설명하고, 모든 문장은 끝까지 완성한다.
- 자막 시작은 대응하는 화면 또는 로그 박스보다 약 0.12초 늦게 배치한다.
- 자막 때문에 버튼이나 로그가 가려져도 자막 위치를 옮기지 않는다. 브라우저 화면을 위로 줄여 하단 전용 여백을 확보한다.

## 4. 검토용 전체 자막 목록

아래 문장만 순서대로 사용한다. 영상 편집 전에는 문구를 임의로 축약하지 않는다.

> v10 Agent 흐름 추가 문구: 7번 다음에 `초안생성 버튼을 누르면 위험성평가 초안 작성 Agent가 먼저 실행됩니다.<br>이어 검토 Agent가 초안을 보완하고 고위험 항목은 조치계획 Agent가 처리합니다.`를 표시한다. 각 실제 시작 로그에서는 차례로 `공통 근거가 준비되면 위험성평가서 초안 작성 Agent가 실행됩니다.`, `이제 위험성평가서 초안을 기반으로 검토 Agent를 실행합니다.<br>원본 10개 조합과 원인·결과·근거 및 위험도 과소평가 여부를 다시 확인합니다.`, `검토가 끝나면 고위험 항목을 대상으로 조치계획 Agent를 실행합니다.`를 표시한다. 이 네 문구는 아래 기존 설명보다 우선한다.

1. 먼저 설비 식별정보와 운전 조건을 빈 화면에서 직접 입력합니다.<br>이 정보는 Agent가 원인·결과·안전조치를 판단할 때 함께 사용하는 업무 맥락입니다.
2. 유사 HAZOP 문서 ID는 과거 표준 문서를 검색하는 기준입니다.
3. Gas Cabinet 누출, MFC 과다 유량, Purge 시작 실패 이력을 입력합니다.<br>이 사고·정비 이력은 위험도 빈도와 담당자 확인 항목을 판단하는 근거로 연결됩니다.
4. ASM Epsilon3200 Excel에는 4개 Node와 10개 Guideword 평가 조합이 준비되어 있습니다.<br>이제 작성된 #1 노드리스트와 #2 가이드워드를 시스템에 업로드합니다.
5. Excel을 업로드하면 시스템이 4개 Node와 10개 평가 조합을 읽어 AI가 평가대상을 새로 만들지 않도록 입력 범위를 고정합니다.
6. Node별 취급 범위에 맞는 물질을 입력합니다.<br>입력된 물질명은 KOSHA MSDS와 누출·화재·폭발 위험 근거를 찾는 검색어가 됩니다.
7. 입력값과 Excel 내용을 확인한 뒤 AI 초안생성을 실행합니다.<br>초안생성 클릭 후 실제 Agent 로그를 실시간으로 확인해보겠습니다.
8. 왼쪽에는 #3·#4 작성 진행상태가 보이고 오른쪽에는 전체 Agent 실행 기록이 표시됩니다.
9. 실제 로그 확대 버튼을 눌러 핵심 로그를 읽기 쉽게 확인합니다.
10. Planning 단계는 실행 전에 해야 할 일을 순서대로 나누는 단계입니다.<br>현재 입력과 Excel을 처리할 작업 계획을 먼저 확정합니다.
11. 현재 Excel 원본에는 4개 Node와 10개 Node·변수·Guideword 평가 조합이 있습니다.<br>Planning 로그는 이 10개 원본 조합을 빠짐없이 처리할 계획을 세웠다는 뜻입니다.
12. Planning 실행 현황에는 각 작업의 대기·진행·완료 상태가 기록됩니다.
13. Skill 로그는 Agent가 HAZOP 작성 규칙과 빈도·강도 판단 기준을 읽고 적용한 기록입니다.<br>쉽게 말하면 Agent가 작업 전에 따라야 할 안전 평가 지침을 확인하는 단계입니다.
14. Workflow는 세 물질의 MSDS와 사고·정비 이력, 연결된 표준 HAZOP 문서를 먼저 조회합니다.<br>이 자료는 작성·검토·조치계획 Agent가 함께 사용하는 공통 근거가 됩니다.
15. Tool Call은 추가 자료가 필요할 때 Agent가 어떤 검색어와 물질명으로 조회했는지 남기는 기록입니다.<br>사고이력, 유사 HAZOP, KOSHA MSDS 조회가 실제 작업 흐름 안에서 이어집니다.
16. Tool 결과에는 검색된 문서 ID와 물질별 MSDS 출처가 남습니다.<br>따라서 결과 문장의 근거가 어디에서 왔는지 로그와 결과표에서 추적할 수 있습니다.
17. 작성 Agent는 Excel 조합과 조회 근거를 하나의 Context로 결합합니다.<br>10개 조합마다 일탈·원인·결과·현재 안전조치와 빈도·강도 후보를 작성합니다.
18. Self-Correction은 첫 초안을 그대로 제출하지 않고 스스로 다시 검토하는 단계입니다.<br>근거 부족, 입력 제약 위반, 위험도와 조치 대상 불일치를 찾아 보완합니다.
19. 별도의 검토 Agent는 원본 10개 조합이 그대로 유지됐는지 다시 대조합니다.<br>원인과 결과의 연결, MSDS 근거, 위험도 과소평가 여부를 독립적으로 확인합니다.
20. 수정 전·후 내용과 수정 이유는 Self-Correction 로그에 함께 남습니다.<br>자동 보완이 어려운 판단은 숨기지 않고 담당자 확인 필요 항목으로 넘깁니다.
21. 검토 결과가 반영되면 시스템이 빈도 곱하기 강도로 전체 위험도를 다시 계산합니다.<br>최종 위험도 9 이상인 항목만 고위험 조치 대상으로 선별합니다.
22. 조치계획 Agent는 선별된 고위험 항목만 전달받습니다.<br>현재 안전조치의 부족점을 확인하고 추가 개선대책과 조치 후 잔여위험을 작성합니다.
23. 최종 화면에서도 왼쪽의 #3·#4 결과와 오른쪽의 전체 Agent 로그를 함께 확인할 수 있습니다.

### 배속과 화면 움직임 규칙

- 0.5초 단위 프레임 점프, 정지 화면 반복, 로그가 순간이동하는 편집은 금지한다.

- 강조 로그는 생성 순서대로 보여주며, 이미 지나간 과거 단계로 다시 올라가지 않는다.
- 자막 자체에는 속도·줌·잔상 효과를 적용하지 않는다.

## 7. 로그 화면 촬영 기준

### 반드시 보여줄 로그

1. 입력 조합 검증과 10행 고정.
2. `write_todos`로 생성된 Planning과 완료 현황.
3. 적용 Skill 이름과 Skill 본문 적용 확인.
4. MSDS, 사고·정비 이력, 표준 HAZOP 조회 입력과 출처.
5. Context가 #3 작성 Agent에 전달되는 로그.
6. 시스템의 `빈도 × 강도` 계산과 범위 검증.
7. 별도 검토 Agent의 Self-Correction 대표 변경.
8. 위험도 9 이상 실제 2~4건 선별.
9. 조치계획 Agent Planning과 #4 생성 완료.

### 로그 강조 방식

- 로그 강조색은 첨부된 v3 예시처럼 청록색 또는 초록색으로 고정한다.
- 강조 대상 로그 바깥쪽에 4~6px 둥근 초록 테두리만 표시하고 내부 배경색과 그림자는 사용하지 않는다.
- 한 번에 하나의 핵심 로그만 강조한다. 여러 박스를 동시에 켜서 시선을 분산시키지 않는다.
- 강조하는 단일 실행 로그 박스는 가능한 한 화면 수직 중앙에 오도록 스크롤한다. Agent 전체 카드에는 테두리를 두르지 않는다.
- 강조 로그를 오래 정지해 설명하지 않는다. 필요한 경우에만 속도를 잠깐 낮춘다.
- 빨간 테두리와 빨간 반투명 배경은 사용하지 않는다.

### 설명하면 안 되는 내용

- 여러 후보 계획을 동시에 비교하는 ToT가 실제 구현된 것처럼 설명하지 않는다.
- 모델 내부의 숨은 생각을 화면에 공개한다고 설명하지 않는다.
- 현재 구현은 고정된 안전 Workflow 안에서 Agent가 명시적 작업 목록을 만들고, 필요한 근거를 조회하고, 별도 Agent가 결과를 다시 검토하는 구조라고 설명한다.
- 단순히 `Thinking → Tool Call`이라고만 설명하지 않는다. 각 로그가 어떤 입력을 확인하고 다음 단계에 무엇을 전달하는지 자막으로 연결한다.

## 9. ASM 10행 샘플과 고위험 2~4건 허용 기준

### 입력 평가 조합

| No | Node | 변수 | Guideword | 예상 위험 성격 |
|---:|---|---|---|---|
| 1 | Gas Cabinet | Containment | Leak | 고위험 목표 |
| 2 | Gas Cabinet | Containment | Rupture | 중위험 목표 |
| 3 | VMB 및 공급 배관 | Flow | Reverse | 중위험 목표 |
| 4 | MFC 유량 제어 구간 | Flow | More | 고위험 목표 |
| 5 | MFC 유량 제어 구간 | Flow | Less | 중위험 목표 |
| 6 | Purge 및 Scrubber 구간 | Purge | No | 고위험 목표 |
| 7 | Gas Cabinet | Flow | No | 저·중위험 목표 |
| 8 | VMB 및 공급 배관 | Pressure | Less | 저·중위험 목표 |
| 9 | MFC 유량 제어 구간 | Signal | No | 저·중위험 목표 |
| 10 | Purge 및 Scrubber 구간 | Purge | More | 저·중위험 목표 |

최근 6행 실제 실행에서 고위험 3건은 `Leak`, `MFC Flow/More`, `Purge/No`였으므로 이 조합을 그대로 유지했다. 촬영 입력에는 이 세 조합의 경보·Near Miss 이력을 명시하고, 새 4행은 발생 이력이 없다고 구분했다.

현재 main 소스의 DeepAgent에서 Node별 물질을 0·1·2·3종으로 다르게 입력해 사전 실행한 결과는 위험성평가 10건, 고위험 3건, 조치계획 3건이었다. 고위험은 `Gas Cabinet/Containment/Leak`, `MFC/Flow/More`, `Purge/Purge/No`이며 모두 빈도 3 × 강도 4 = 위험도 12로 계산됐다. 촬영 직전에도 같은 입력으로 한 번 더 재검증한다.

다만 위험도는 Excel에 미리 넣는 값이 아니다. Agent가 실행 중 제안한 빈도와 강도를 시스템이 계산하므로 실제 촬영 결과가 2~4건이면 정상으로 인정한다. 결과 숫자는 편집으로 3건에 맞추지 않고, 화면과 마지막 자막 모두 실제 건수를 사용한다.

- 일반 주소 `/`에서는 로컬 테스트가 편하도록 ASM Epsilon3200 값이 실제 기본값으로 입력되어 Agent에게 전송된다.

## 12. 최종 강조 로그 중앙 배치 및 도입 자막 순서

### 최종 고정 사항

1. 강조한 로그는 애플리케이션의 높이·padding·margin을 변경하지 않고 `.log-box.scrollTop`만 조정해 화면 중앙에 둔다.
2. 마지막 로그처럼 아래에 내용이 없어 중앙 스크롤이 불가능한 경우, 다음 실제 로그가 쌓인 뒤 이전 로그로 스크롤해 박스 전체를 보여준다.
3. Self-Correction 시작 로그는 완료·고위험 선별 로그가 실제로 쌓인 뒤 다시 해당 위치로 스크롤해 강조한다.
4. Agent 실행 도입부 자막은 다음 세 cue 순서로 고정한다.
   - `왼쪽에는 #3·#4 작성 진행상태가 보이고 오른쪽에는 전체 Agent 실행 기록이 표시됩니다.`
   - `기본적인 시스템의 흐름은 초안 작성 Agent가 먼저 실행됩니다.` / `이어 검토 Agent가 초안을 보완하고 고위험 항목은 조치계획 Agent가 처리합니다.`
   - `로그 확대 버튼을 눌러 핵심 Agent 실행 기록을 읽기 쉽게 확인합니다.`
5. 위 세 cue 외의 자막 내용은 기존 검토 확정본을 유지한다.
