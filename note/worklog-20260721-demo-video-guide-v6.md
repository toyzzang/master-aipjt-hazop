# 작업 설계: ASM 10행 시연 샘플과 최종 촬영 가이드 현행화

## 요청 배경

- 시연영상 자체는 아직 수정하지 않고, 검토용 자막 전체 문장을 먼저 정리한다.
- 시작 표지에서 `실제 브라우저`, `실제 Excel` 같은 표현을 제거한다.
- 자막은 Agent가 Planning, 근거 조회, 초안 작성, 독립 검토, 시스템 재검증을 거쳐 결과를 만드는 흐름을 쉽게 설명해야 한다.
- 기존 ASM Epsilon3200 시연 시나리오는 유지하되, 가이드워드 입력을 약 10행으로 늘리고 최종 고위험 조치 대상은 3건이 되도록 구성한다.

## 작업 범위

1. 기존 ASM Excel 양식과 최근 6행 실행 결과를 확인한다.
2. 기존 3개 고위험 조합은 유지하고, 낮거나 중간 위험으로 판단될 4개 조합을 추가한 10행 샘플을 별도 파일로 만든다.
3. `#1 노드리스트`, `#2 가이드워드`만 입력하고 `#3 위험성평가`, `#4 조치계획서` 데이터 영역은 비워 둔다.
4. 3분 후반~4분 초반 촬영 흐름, 화면 동작, 전체 자막, 강조 로그, 결과 검수 기준을 하나의 Markdown 가이드로 현행화한다.

## 가정과 제한

- 위험도는 Excel 입력값이 아니라 실행 중 Agent가 제안한 빈도·강도를 시스템 코드가 곱해 계산한다.
- 따라서 입력 파일만으로 고위험 3건을 수학적으로 고정할 수는 없다. 최근 실제 실행에서 고위험이었던 3개 조합은 그대로 두고, 추가 4행은 저·중위험 성격으로 설계한다.
- 최종 촬영 전 사전 실행에서 고위험 3건을 확인하며, 다를 경우 결과를 편집으로 조작하지 않고 입력 근거와 시나리오를 조정해 다시 실행한다.
- 기존 사용자 파일은 덮어쓰지 않고 새 파일명으로 제공한다.

## 성공 기준

- 새 ASM 파일의 가이드워드 입력 행은 정확히 10개다.
- `#3`, `#4`에는 헤더 외 결과 데이터가 없다.
- 기존 실제 실행의 고위험 3개 조합인 `Leak`, `MFC Flow/More`, `Purge/No`가 유지된다.
- 가이드에는 시작 표지와 3개 목차 표지, 고정 하단 자막, 전체 로그, 클릭 표시, 결과 그리드 강조, 종료 표지가 반영된다.
- 요청 문구 `클릭 후 실제 Agent 로그를 실시간으로 확인해보겠습니다.`가 그대로 포함된다.

## 검증 방법

- Spreadsheet artifact 도구로 모든 시트를 렌더링해 레이아웃을 확인한다.
- 주요 셀 범위와 수식 오류를 검사한다.
- 애플리케이션의 Excel 미리보기 로직으로 Node 수와 Guideword 수를 확인한다.
- 자막은 한 문장이 불필요하게 둘로 끊기지 않는지, 모든 문장이 하단 중앙의 동일한 스타일로 표시 가능한 길이인지 점검한다.

## 검증 결과

- 애플리케이션 미리보기/검증 로직에서 Node 4건, Guideword 10건을 정상 인식했다.
- `#3 위험성평가`와 `#4 조치계획서`는 헤더만 있고 데이터 Row는 비어 있다.
- 전체 시트를 렌더링해 기존 양식과 열 너비가 유지된 것을 확인했고 수식 오류는 0건이다.
- 현재 main 소스의 DeepAgent 사전 실행 결과는 위험성평가 10건, 고위험 3건, 조치계획 3건이다.
- 고위험 3건은 `Gas Cabinet/Containment/Leak`, `MFC/Flow/More`, `Purge/Purge/No`이고 각각 위험도 12다.

## 2차 자막·촬영 피드백 반영

- 모든 자막을 하단 중앙의 전용 여백, 동일 크기, 동일 디자인으로 고정했다.
- 시작 표지 금지 문구에 `실제 브라우저`, `실제 엑셀`, `실제 에이전트 로그 시연`을 명시했다.
- 표지·목차·종료 화면은 v3의 짙은 남색, 흰색, 청록색 고정 디자인을 사용한다.
- Agent 실행 중 화면을 정지하지 않고 스피너와 로그 갱신을 유지하며, 대기 구간만 자연스럽게 배속한다.
- 로그와 결과 그리드의 강조색을 빨간색이 아닌 초록색으로 고정했다.
- 종료 표지는 이름과 사번 없이 `시연을 마치겠습니다`, `감사합니다.`를 표시한다.
- 전체 권장 길이를 3분 58초로 조정하고 유사 HAZOP 문서 ID 설명은 2초로 제한했다.
- 자막 4번의 기존 문장을 제거하고, 5번과 7번은 사용자가 지정한 문장으로 교체했다.
- Planning과 Self-Correction 설명을 실제 실행 계획, 독립 검토, 시스템 재계산 흐름에 맞게 보완했다.

## 제출 가이드 최종 대조 및 3차 보완

- 첨부된 `AI_Master_Project_시연_영상_제출_가이드 (1).docx`의 3쪽을 모두 검토했다.
- 필수 평가 요소인 수행 시나리오 소개, Planning, Tool Call, Self-Correction, 핵심 터미널·콘솔, 최종 결과 확인을 촬영 타임라인과 일대일로 대조했다.
- 브라우저 UI만 보여주는 인상을 피하기 위해 실제 백엔드 `AGENT_TRACE`를 약 8초 보여주는 구간을 추가했다.
- 결과의 완성도뿐 아니라 처리 성과를 짧게 확인할 수 있도록 `입력 10 → #3 10 → 고위험 3 → #4 3` 정량 요약을 추가했다.
- 결과 확인은 43초로 확보하고, 전체 권장 길이는 4분 03초로 조정했다.
- 제출 규격은 MP4, 500MB 이하, `10346_김창은_시연영상.mp4`로 체크리스트에 고정했다.
- Node별 물질 입력은 실제 업무처럼 Gas Cabinet 2종, VMB 0종, MFC 1종, Purge 및 Scrubber 3종으로 다르게 구성했다.
- ToT나 모델의 숨은 생각을 실제 구현처럼 과장하지 않고, 화면에서 검증 가능한 계획·도구 호출·독립 검토 로그만 설명하도록 명시했다.
- 변경된 Node별 물질 입력으로 현재 main의 DeepAgent를 다시 실행했고, `deepagent` 모드에서 #3 10건, 고위험 3건, #4 3건이 생성되는 것을 확인했다.
- 재검증된 고위험 3건은 `Gas Cabinet/Containment/Leak`, `MFC/Flow/More`, `Purge/Purge/No`이며 각각 빈도 3 × 강도 4 = 위험도 12다.

## 4차 최종 검토 반영 설계

### 변경 범위

1. 촬영 가이드에서 실제 백엔드 터미널 장면을 제거하고, 브라우저의 확대된 전체 Agent 로그만 사용한다.
2. 최종 결과 구간은 소스에 별도 요약 UI를 추가하지 않고 기존 #3·#4·검토표를 촬영 편집의 초록 테두리와 하단 자막으로 설명한다.
3. 최종 영상은 동일 원본을 기준으로 자막 포함본과 무자막본 두 가지로 출력하도록 가이드에 고정한다.
4. 최초 브라우저 로딩 시 Maker, Model, 유사 HAZOP ID, 운전 의도, 사고·정비 이력을 실제 값이 없는 빈 상태로 만든다.
5. 빈 입력란에는 ASM Epsilon3200 시연값을 placeholder로만 보여준다. placeholder는 실제 전송값이 아니다.
6. Node별 물질 placeholder는 Gas Cabinet 2종, VMB 입력 없음, MFC 1종, Purge 및 Scrubber 3종으로 맞춘다.

### 성공 기준

- 새로고침 직후 모든 기본정보 입력값은 빈 문자열이다.
- 화면에는 ASM Epsilon3200 예시 placeholder가 표시된다.
- 사용자가 입력하지 않은 placeholder 문구는 작업 요청에 포함되지 않는다.
- 촬영 가이드에는 백엔드 터미널 장면이 남아 있지 않다.
- 자막 포함본과 무자막본의 화면·길이·편집은 동일하고 자막 레이어 유무만 다르다.

### 4차 검증 결과

- TypeScript typecheck와 Vite production build가 모두 통과했다.
- 로컬 서버와 실제 브라우저에서 Maker, Model, 유사 HAZOP 문서 ID, 운전 의도, 사고·정비 이력의 값이 모두 빈 문자열인 것을 확인했다.
- 같은 화면에서 ASM, Epsilon3200, 표준 HAZOP ID, 운전 의도, 사고 이력 예시가 placeholder로 표시되는 것을 확인했다.
- 영상용 Excel을 로컬 업로드 API로 읽어 4개 Node와 10개 Guideword 평가 조합이 반환되는 것을 확인했다.
- 촬영 가이드에서 백엔드 터미널 장면을 제거하고 브라우저의 확대된 전체 Agent 로그로 대체했다.

## 5차 실제 영상 제작 설계

### 기본값과 촬영 모드 분리

- 일반 접속 `/`에서는 ASM Epsilon3200의 Maker, Model, 문서 ID, 운전 의도, 사고·정비 이력을 실제 기본값으로 넣는다.
- 일반 접속에서 ASM 10행 Excel을 업로드하면 Node별 물질도 2종·0종·1종·3종 실제 기본값으로 채운다.
- 촬영 접속 `/?recording=1`에서는 같은 입력을 모두 빈값으로 시작해 Key-in 과정을 녹화한다.
- placeholder는 두 모드 모두 유지하되 촬영 모드에서만 빈 화면의 입력 안내 역할을 한다.

### 영상 산출물

- 실제 브라우저에서 10행 Excel 업로드, Agent 전체 로그, #3·#4·검토표를 녹화한다.
- 백엔드 터미널은 영상에 넣지 않는다.
- 원본 촬영과 편집 타임라인은 하나로 유지하고 자막 레이어 유무만 달리해 두 파일을 출력한다.
- 자막 포함본과 무자막본 모두 3분 50초~4분 05초, 1920×1080, MP4, 500MB 이하로 만든다.

### 실제 제작 및 최종 검증 결과

- 현재 브랜치 소스로 로컬 서비스를 실행하고 `/?recording=1`의 빈 입력 화면부터 실제 브라우저 조작을 녹화했다.
- Maker·Model·운전 의도·사고 이력을 직접 Key-in한 뒤 실제 10행 Excel을 업로드했다.
- Node별 물질은 Gas Cabinet 2종, VMB 없음, MFC 1종, Purge 및 Scrubber 3종으로 서로 다르게 입력했다.
- AI 초안생성 후 전체 Agent 로그가 실시간으로 쌓이는 화면과 확대 로그의 Planning·Skill·Tool Call·Context·Self-Correction을 담았다.
- 결과는 실제 화면의 #3 위험성평가 10건, 고위험 3건, #4 조치계획 3건, 초안 검토 및 보완 내역을 천천히 스크롤해 확인했다.
- 중요 로그와 결과 그리드는 초록색 테두리로 강조했고 마우스 포인터와 초록색 클릭 표시를 포함했다.
- 대기 구간만 연속 배속하고 입력·로그 설명·결과 확인 구간은 정상 속도로 유지했다.
- 모든 자막을 1920×1080 화면 하단 중앙의 동일한 28px 흰색 글씨와 반투명 진회색 박스 스타일로 고정했다.
- 최초 표지, 3개의 목차 표지, 마지막 `시연을 마치겠습니다` 표지를 짙은 남색·흰색·청록색 디자인으로 통일했다.
- 최종 종료 표지가 잘리지 않도록 마지막 구간을 다시 편집했고 약 4초 동안 유지되는 프레임을 확인했다.
- 자막 포함본과 무자막본은 모두 243.84초(약 4분 04초), 1920×1080 MP4로 생성했다.
- 자막 포함본의 하단 자막 배경과 종료 표지를 실제 추출 프레임으로 최종 확인했다.

## 6차 영상 피드백 반영 설계

### 변경 범위

1. 하단 자막을 한 번에 한 문장만 표시하고, 흰색 38px 글씨와 화면 폭에 가까운 반투명 진회색 고정 바를 사용한다.
2. Excel의 #1·#2 시트를 별도 이미지로 보여주는 장면은 삭제하고 실제 업로드 동작만 남긴다.
3. 스피너·로그 이동·로그 갱신처럼 화면만 보아도 알 수 있는 동작 설명은 자막에서 제거한다.
4. 확대 로그는 공통 MSDS 근거 조회, DeepAgent 시작, Planning, Skill, Tool, Context, Self-Correction, 조치계획 순서로만 보여준다.
5. 과거 로그로 다시 올라갔다가 현재 로그로 돌아오는 편집을 제거한다.
6. 중요 로그는 청록색 굵은 둥근 테두리와 매우 옅은 청록색 배경으로 강조한다.
7. 마우스 클릭 표시는 기존보다 진한 초록색 `#006b4f`과 7px 테두리를 사용한다.

### 성공 기준

- 자막이 겹치지 않고 모든 자막이 한 줄로만 표시된다.
- 자막 글씨가 이전 영상보다 크고 하단 고정 바의 중앙에 배치된다.
- Excel 내용을 캡처한 별도 화면이 영상에 없다.
- 확대 로그에서 이전 단계로 역행하는 장면이 없다.
- 중요 로그의 테두리와 클릭 표시가 첨부된 참고 이미지와 유사한 진한 청록색으로 보인다.

### 6차 실제 제작 및 최종 검증 결과

- 현재 브랜치의 실제 브라우저와 Agent를 다시 실행해 영상 원본을 새로 녹화했다.
- 실제 실행 결과는 #3 위험성평가 10건, 위험도 9 이상 조치계획 3건, 초안 검토 및 보완 내역 2건이다.
- Agent 오류나 임시 대체 결과 없이 정상 완료된 실행만 최종 영상에 사용했다.
- Excel 시트 내용을 별도 캡처한 장면은 모두 제외하고, 브라우저에서 파일을 업로드하는 실제 동작만 남겼다.

- 중요 로그는 5px 진한 청록색 둥근 테두리와 옅은 청록색 배경으로 강조했다.

## 9차 시작 프레임·원본 로그 여백·3-Agent 흐름 보완 설계

### 변경 범위

1. 원본 영상 첫 장면은 시작 표지가 완전히 그려진 프레임부터 사용한다. 브라우저 초기 화면이나 표지 생성 전 프레임을 한 장도 포함하지 않는다.
2. 촬영용으로 추가했던 `.log-box`의 `padding-bottom: 430px`을 제거한다. 로그 영역의 크기와 여백은 현재 애플리케이션 소스의 기본값을 그대로 사용한다.
3. 강조 로그를 중앙에 배치할 때 DOM 크기나 여백을 바꾸지 않고 `.log-box.scrollTop`만 조정한다. 마지막 로그는 원본 스크롤 한계 안에서 가장 잘 보이는 위치까지만 이동한다.
4. AI 초안생성 직후 전체 흐름을 `위험성평가 초안 작성 Agent → 검토 Agent → 조치계획 Agent` 순서로 한 번 설명한다.
5. `risk-draft-agent (위험성평가 초안 작성) Agent를 실행합니다.` 로그가 보이는 시점에 `위험성평가서 초안 작성 Agent가 실행됩니다.` 자막을 표시한다.
6. `risk-review-agent (초안 검토 및 보완) Agent를 실행합니다.` 로그가 보이는 시점에 `이제 위험성평가서 초안을 기반으로 검토 Agent를 실행합니다.` 자막을 표시한다.
7. `action-plan-agent (고위험 조치계획 작성) Agent를 실행합니다.` 로그가 보이는 시점에 `검토가 끝나면 고위험 항목을 대상으로 조치계획 Agent를 실행합니다.` 자막을 표시한다.
8. 각 시작 자막 이후에 해당 Agent의 Planning, Skill, Tool, Self-Correction 등 기존 상세 설명을 이어간다.
9. 기존 v9를 덮어쓰지 않고 `real-browser-demo-v10-agent-flow` 폴더와 v10 파일명을 사용한다.
